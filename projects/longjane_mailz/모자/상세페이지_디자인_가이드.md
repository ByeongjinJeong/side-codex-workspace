# MAILZ Hat Detail Page Guide

## Core Intent

The page must sell the product through a clear visual argument:

1. Catch the customer's problem fast.
2. Prove the product feature with real photos.
3. Keep the text hierarchy strong enough to feel like a detail page, not a report.
4. End with trustworthy purchase, spec, and care information.

This guide is written for the agent building the page. Use the numeric ranges as working ranges, not rigid code constants. Adjust by sentence length, photo crop, section density, and reference rhythm.

Korean copy should remain Korean. Section names and example phrases below are intentionally Korean because the final page is Korean.

## Canvas

- Page width: `1500px`
- Main content width: `1360px`
- Side margin: `70px`
- Default full photo width: `1360px`
- Default feature photo height: `1000px-1200px`
- Hero photo height: `1360px-1520px`
- Two-column photo: about `660px-680px` wide each
- Four-grid photo: about `670px x 700px`
- Section background bands may be white, very light gray, or very light pink.

Do not make the page feel like a web app. No cards inside cards, no app-like controls, no navigation UI.

## Color

- Primary pink: `#E0335C`
- Soft pink background: `#FFE8F0`
- Text black: `#050505`
- Muted body text: `#5C5752`
- Canvas white: `#FFFFFF`
- Light section gray: `#F0F0EE`
- Card gray: `#F6F6F6`
- Hairline: `#E8E2DE`

Pink is a hook, not a wallpaper. Use it for title bars, labels, stamps, and small accents. If every section is equally pink, the page looks cheap.

## Font Policy

Use accessible placeholder fonts only.

- Title, section title, customer voice, stamp, and photo label placeholder: `NanumGothic ExtraBold`
- Description and body: `NanumGothic Bold`
- Specs, captions, legal/care text: `NanumGothic Regular` or `NanumGothic Bold`

Do not set inaccessible fonts as the guide default. If a different title font is desired later, replace only the placeholder title/label styles in Figma.

Important Figma rule:

- Text that visually appears in Figma may still be missing for plugins.
- If `hasMissingFont` is true, do not rely on that font for automated edits.
- Build with accessible placeholder styles first. Final font replacement can happen manually through Figma style replacement.

## Typography Rhythm

Use relative hierarchy first, exact pixels second.

The detail page should feel closer to the reference than to a report:

```text
big hook
supporting title
short body
large proof photo
short label
```

Recommended working ranges for a 1500px-wide page:

| Role | Font | Size Range | Line Height | Notes |
|---|---|---:|---:|---|
| Hero title | NanumGothic ExtraBold placeholder | 120-132 | 1.05-1.10 | First-screen strongest message |
| Pink bar title | NanumGothic ExtraBold placeholder | 92-112 | 1.05-1.12 | Must sit inside the pink bar |
| Feature title | NanumGothic ExtraBold placeholder | 92-112 | 1.10-1.18 | Use 88-96 if two lines are long |
| Feature description | NanumGothic Bold | 46-58 | 1.35-1.50 | Support text, not another title |
| Feature body/caption | NanumGothic Bold | 38-46 | 1.40-1.55 | Small explanatory copy |
| Customer header | NanumGothic ExtraBold placeholder | 86-100 | 1.08-1.15 | `Customer PICK` block title |
| Customer voice | NanumGothic ExtraBold placeholder | 54-66 | 1.18-1.28 | Customer concern, emotional |
| Customer answer | NanumGothic Bold | 34-42 | 1.35-1.50 | Product answer, calmer |
| Benefit card title | NanumGothic ExtraBold placeholder | 56-70 | 1.10-1.20 | Large, but must fit 3 columns |
| Benefit card body | NanumGothic Bold | 34-42 | 1.35-1.50 | Do not compete with title |
| Photo label | NanumGothic ExtraBold placeholder | 48-62 | 1.05-1.12 | 2-5 words only |
| Spec/care table | NanumGothic Regular/Bold | 28-34 | 1.35-1.55 | Utility text |

Do not blindly shrink all text to avoid collision. Keep hooks large, then solve collisions with better section spacing, shorter line breaks, or localized size adjustments.

## Flexible Sizing Rules

Use these decisions in order:

1. Classify the text role by node name and section purpose, not by its current font size.
2. If a hook feels weak compared with the reference, increase the hook first.
3. If text collides, first check whether the section spacing or y-position is wrong.
4. Only reduce font size when the sentence itself is too long for the role.
5. Do not let descriptions become visual titles.
6. Do not let labels become paragraphs.

Hard safety rules:

- Pink bar text must be vertically centered inside the bar.
- Text below a pink bar must never touch or overlap the bar.
- Text must not invade photos, tables, labels, or section backgrounds.
- 3-column benefit card text must fit inside each card without clipping.
- Text overlap checks must include text-vs-shape relationships, not just text-vs-text.

## Page Flow

Recommended order:

1. Hero
2. Customer PICK
3. Three benefit cards
4. 앞·옆·뒤 360도 커버
5. 긴 뒷챙
6. 똑딱이 고정
7. 스토퍼 턱끈
8. 턱끈 분리
9. 데일리핏
10. 컬러 라인업
11. 제품 상세 정보
12. 세탁 및 관리 안내

This is a persuasion sequence, not a data dump. Problem first, proof second, purchase confidence last.

## Section Patterns

### Hero

Purpose: make the main purchase reason obvious in the first screen.

Recommended structure:

- Small store/brand line
- Black hero title
- Pink hook bar
- Large real wearing photo
- Optional small supporting cut
- Small product name caption

Example copy:

```text
얼굴부터 뒷목까지
360도 햇빛 철벽 커버
```

Rules:

- The product photo must be dominant.
- Do not add long body copy in the hero.
- Decorative dots are optional and should not compete with the product.

### Customer PICK

Purpose: show that the page understands the customer's worries.

Recommended structure:

- `Customer PICK`
- `고객님의 이런 의견을 듣고, 엄선했어요`
- Three customer concerns
- Pink `해결` stamp per concern
- Short product answer under each concern
- Four real wearing photos

Example:

```text
고민: 얼굴은 물론 뒷목까지 햇빛에 타는 게 걱정돼요
해결: 앞·옆·뒤로 이어지는 빅챙이 얼굴 라인부터 목 뒤까지 넓게 커버해요
```

Rules:

- Concerns should feel like a customer voice.
- Answers should be smaller and calmer than concerns.
- The stamp may rotate slightly, but it must not cover the text.
- Do not skip the product answer.

### Three Benefit Cards

Purpose: compress the product value into three clear reasons.

Suggested benefits:

```text
앞·옆·뒤 빅챙
똑딱이 뒷챙
분리형 스토퍼 턱끈
```

Rules:

- Three cards should align as a single row.
- Card titles are large; card bodies are smaller.
- If a body line is long, rewrite or split it. Do not let it fill the whole card.

### Feature Section

Purpose: prove one product feature with one visual argument.

Recommended structure:

- Pink bar hook
- Black feature title
- Short muted description
- Large real photo
- Short photo label or measurement label

Example:

```text
앞·옆·뒤 360도 커버
햇빛이 강한 날, 얼굴만 가려서는 부족하니까
정면은 물론 옆 라인과 뒷목까지 넓은 챙으로 자연스럽게 감싸줘요
```

Rules:

- One feature section should have one main message.
- The photo is the proof; it must be visually stronger than the body copy.
- A label should point to what the customer should notice.

### Color Lineup

Purpose: show available colors and make choice easy.

Recommended structure:

- Pink bar: `블랙/화이트/베이지`
- Main title: `3가지 색상 선택 가능`
- Short coordination copy
- Large color photo
- Color names/captions only if they clarify the image

Rules:

- Keep this simpler than feature sections.
- Do not invent vague color moods like `야외활동 컬러`.
- The pink title must sit inside the bar. The main title goes below the bar.

### Spec And Care

Purpose: provide purchase confidence and protect against avoidable CS issues.

Required spec content:

- 상품명
- 색상
- 사이즈
- 소재
- 제조국
- 구성
- 턱끈 탈부착 여부
- 뒷챙 고정 여부
- 사이즈 오차 문구
- 색상 차이 문구

Required care content:

- 세탁기 사용 비권장
- 물세탁/기계세탁/건조기 사용 시 변형 가능
- 오염 부위는 젖은 천으로 가볍게 관리
- 세탁 부주의로 인한 변형/이염/수축/부자재 손상은 교환 및 반품 제한 가능

Rules:

- Spec and care sections should be quieter than selling sections, but not broken or hidden.
- Pink bar title still needs correct alignment.
- Tables and care boxes should look organized, not like leftover text.

## Photo Rules

- Use real product photos as evidence.
- Prefer large full-width photos for feature proof.
- Use two-column photos only when comparison is needed.
- Use four-grid photos for Customer PICK or wearing examples.
- Do not expose black masks, empty rectangles, or broken crops.
- For vertical wearing photos, crop confidently instead of forcing them into awkward squares.
- Labels should sit on photos only when they clarify the feature.

## Copy Mapping

Confirmed copy should be preserved, but each sentence must be assigned to the right visual role:

- Hook: pink bar title or hero title
- Customer problem: customer voice or feature title
- Product answer: description or customer answer
- Measurement/function word: photo label
- Product name/color/spec: caption or table
- Care/legal copy: table or care box

Do not put a long explanatory sentence into a title role just because it is important.

## QA Checklist

Run this before calling a frame complete:

- [ ] The page is 1500px wide and main photos use the 1360px content rhythm.
- [ ] Each section has one clear message.
- [ ] Hooks feel strong enough compared with the reference.
- [ ] Body copy is visibly secondary to titles and photos.
- [ ] Pink bar text is centered inside the bar.
- [ ] Text below a pink bar does not touch the bar.
- [ ] Text does not overlap other text.
- [ ] Text does not invade photos, labels, tables, or background blocks.
- [ ] Customer PICK has 3 concerns and 3 answers.
- [ ] The three benefit cards align and read as a set.
- [ ] Feature photos prove the exact feature being claimed.
- [ ] Photo labels are short, usually 2-5 words.
- [ ] Color lineup is clean and not over-explained.
- [ ] Spec and care sections look intentionally designed, not dumped.
- [ ] No inaccessible font is required for automated editing.
- [ ] All text remains editable with available placeholder fonts.

## Failure Patterns To Avoid

- Treating the guide like a fixed CSS file.
- Shrinking every text block just to avoid overlap.
- Making all text the same weight and size.
- Letting descriptions become titles.
- Letting pink bars float behind misaligned text.
- Checking only text-vs-text overlap and missing text-vs-shape collisions.
- Leaving old experimental styles that confuse future edits.
- Making the lower spec/care sections look broken because they are less visual.
